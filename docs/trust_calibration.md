# Trust-Boundary Calibration (M4)

> _"Formalizing when to trust retrieval versus fallback, and improving
> that boundary."_ — the internship JD bullet, verbatim.

The refusal gate is the boundary that separates "answer this with the
RAG context" from "refuse this — the context isn't sufficient." It has
exactly one tunable knob: `RELEVANCE_THRESHOLD`, the minimum max-fused
retrieval score below which the gate refuses pre-LLM. This document
defends the chosen value with a quantitative sweep.

## Methodology

* **Sweep.** `scripts/run_calibration.py` evaluates the refusal gate
  across `RELEVANCE_THRESHOLD ∈ {0.05, 0.075, 0.10, 0.125, 0.15,
  0.175, 0.20, 0.25, 0.30}` (the plan §5.1 grid).
* **Backend-free.** Retrieval is deterministic at a given git sha;
  the gate is a pure function of the retrieved chunks and the
  threshold. No LLM is called, so this sweep costs zero NIM credits
  and reproduces exactly across machines.
* **Dataset.** The 30-example golden set, with 5 queries marked
  `expected_refusal=True` (the OOD cohort) and 25 marked
  `expected_refusal=False` (in-domain).

## Definitions

For each threshold `t` we compute:

```
refusal_precision(t)        = TP / (TP + FP)
refusal_recall(t)           = TP / (TP + FN)
non_refusal_coverage(t)     = TN / (TN + FP)
```

where TP/FP/TN/FN are computed against the per-example
`expected_refusal` ground truth. `refusal_precision` answers "of all
queries we refused, how many should have been refused?";
`refusal_recall` answers "of all queries that should have been
refused, how many did we catch?"; `non_refusal_coverage` answers
"of all in-domain queries, how many were preserved for downstream
generation?"

## Reading the curve

* **Recall is constantly 1.000** across every threshold tested. The
  OOD cohort's max retrieval score is well below the lowest threshold
  in the grid — the lexical-overlap signal cap (described in
  `docs/architecture.md` §Retrieval) zeroes them out before the gate
  even reads the score. This is what we want: the gate never misses
  an OOD query in this regime.

* **Precision degrades monotonically** as the threshold rises,
  because in-domain queries with weaker (but legitimate) retrieval
  start tripping the gate. At `t=0.10` the first FP appears: a
  cross-reference lookup whose top chunk scores between 0.075 and
  0.100. By `t=0.30`, 11 in-domain queries are wrongly refused.

* **Non-refusal coverage tracks precision** by construction (TN+FP =
  the in-domain set), so the same story shows up in coverage.

## Defended operating point

**Chosen: `RELEVANCE_THRESHOLD = 0.05`** (the current default).

* `t = 0.05` and `t = 0.075` both achieve perfect precision, recall,
  and coverage on this dataset. They are operationally equivalent
  here.
* We pick `0.05` (the lower of the two) because:
  1. It maximizes the safety margin against future OOD examples that
     happen to score slightly above zero. Picking `0.075` would
     trade a hypothetical safety margin for nothing observable.
  2. The asymmetry in the failure cost matters. A FP (refusing an
     in-domain query) is recoverable — the user reformulates and
     tries again. An FN (silently answering an OOD query) is the
     hallucination scenario this whole system exists to avoid. We
     prefer the conservative endpoint.
* Below `t = 0.05` (e.g. `t = 0.025`), recall would still be 1.000
  on this dataset but the OOD set's max scores hit zero exactly, so
  the gate becomes degenerate (a trivial floor of "refuse if no
  chunks at all" — the lexical-overlap cap and graph expansion alone
  do the work). We document this as a known degenerate band in the
  ADR-005 rationale and don't set thresholds there.

## What this curve does NOT capture

* **The lexical-overlap cap.** The score-cap formula
  `score = fused × top_vec_sim × overlap_frac` (architecture.md
  §Retrieval) does most of the OOD-signal work. The threshold sweep
  measures only the slack in the gate above that cap. A future
  ablation (M3-extension) would zero out the cap to measure the
  threshold's *standalone* discriminating power.
* **Real-LLM faithfulness on no-refusal in-domain queries.**
  FP queries (refused-when-shouldn't-be) drop out before the LLM
  even sees them, so we can't measure how much LLM faithfulness is
  preserved by the threshold choice. Plan M5 (real NIM headlines)
  re-runs the calibration sweep on actual generations and reports
  faithfulness × recall on the in-domain side.
* **Adversarial OOD.** "Tell me about pressure vessels in 1850s
  Britain" is OOD-by-irrelevance but lexically overlaps the
  vocabulary; the current cap might not catch it. We document this
  failure mode in the limitations section of `docs/report.md`.

## Operating-point in code

```python
# src/anvil/generation/refusal_gate.py
RELEVANCE_THRESHOLD: float = 0.05
```

A test in `tests/unit/test_calibration.py` locks this constant to
`0.05` so an accidental edit is caught at CI time. Changing the
operating point is a deliberate process:

1. Re-run `scripts/run_calibration.py` and update
   `data/calibration/calibration.md`.
2. Update this document and the rationale in `docs/design_decisions.md`
   (ADR-005).
3. Update the lock test.
4. Update `docs/test_inventory.md` to reflect the new lock.

## Reproducibility

```bash
uv run python scripts/run_calibration.py
# → data/calibration/calibration.json
# → data/calibration/calibration.md
```

The sweep is deterministic at a given git sha and dataset hash.
Per-example refusal decisions and reasons are recorded in
`calibration.json` (committed) — useful for triaging an unexpected
shift in any single threshold's metrics.
