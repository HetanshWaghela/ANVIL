Pipeline Explainer

How ANVIL processes a query end-to-end. Each step is transparent and auditable.

1

Parse

PDF or Markdown source is parsed into typed DocumentElement objects: sections, tables, formulas, definitions, conditions. Each element has a stable ID, paragraph reference (e.g. A-27), and page number.

↓
2

Index

Parsed elements are indexed three ways: BM25 for keyword matching, vector embeddings for semantic similarity, and a typed knowledge graph (NetworkX) encoding contains/references/requires relationships between sections, tables, and formulas.

↓
3

Retrieve

A query triggers hybrid retrieval: BM25 + vector search are fused via Reciprocal Rank Fusion (RRF), then graph expansion pulls in structurally related elements (e.g. if A-27 is retrieved, Table M-1 and Table B-12 follow via REQUIRES edges). Result: ranked evidence with provenance.

↓
4

Pin

Critical numerical values — allowable stress (Table M-1) and joint efficiency (Table B-12) — come from verified pinned tables, not model memory or RAG extraction. The calculation engine uses Decimal-backed arithmetic on these values. If a value is missing, the system refuses.

↓
5

Calculate

Deterministic formulas compute engineering results (e.g. minimum wall thickness per A-27). Every input, its source, the formula, and the result are recorded as CalculationStep objects with full provenance. The LLM never does arithmetic.

↓
6

Generate

The LLM receives the retrieved evidence and pre-computed calculation results. It produces a structured AnvilResponse with citations, confidence tier, and explanation. Calculation steps are injected by trusted host code, not generated by the model.

↓
7

Validate

The citation enforcer checks every quoted_text against the parsed source document. If a citation cannot be verified (source element missing, quote not found, paragraph mismatch), confidence is downgraded. No ungrounded claims pass through.

↓
8

Refuse

If evidence is insufficient, material data is missing, temperature is out of range, or the query is out-of-domain, the system returns a structured refusal with a clear reason instead of guessing. This is a feature, not a limitation.

Interview context:
• SPES-1 (synthetic standard): full calculation pipeline works end-to-end with pinned tables and deterministic formulas.
• Uploaded real ASME PDF: parser, retriever, graph onboarding work. Retrieval and citation demos are functional.
• Real ASME deterministic calculation requires validated pinned tables, formula tools, and private golden evals — that is ongoing engineering work, not a limitation of the architecture.

Pinned Tables: What, Why, and How

Pinned tables are the most safety-critical design decision in the system. This section explains the reasoning for an interviewer.

What gets pinned?

Data that meets all three criteria:

1. Safety-critical numerical values — values that enter a calculation formula where an error produces a structurally unsafe design. Allowable stress S (temperature-dependent), joint efficiency E (depends on weld type + examination extent), material yield/tensile strengths.

2. Structured tabular form — the data exists as a finite lookup table in the standard, not as free-text prose. Table M-1 (materials), Table B-12 (joint efficiencies), Table UG-43 (pipe threads).

3. Used by deterministic formulas — the value is an input to a host-code calculation (e.g. wall thickness per UG-27/A-27) where the LLM must never do the arithmetic.

What does NOT get pinned?

Prose requirements — "Vessels designed for external pressure shall also satisfy UG-28." This is retrieved by the RAG pipeline and cited. No numerical lookup needed.

Qualitative rules — "Impact testing is required when the minimum design metal temperature is below −29°C." The LLM explains the rule from retrieved context; no calculation.

Design guidance — "Nozzle reinforcement shall be provided per UG-37." The formula tools handle reinforcement calculations using pinned values, but the guidance text itself is retrieved normally.

Onboarding process after PDF upload

When a new standard PDF is uploaded, the system works in two modes:

Immediate (automatic): Parse → Index → Retrieve → Generate. All retrieval, citation, and prose-based Q&A works out of the box. The demo shows this today.

Engineering effort (manual): To enable deterministic calculations for a new standard, an engineer must:

  1. Extract & validate the material property tables from the standard. Cross-check against the published errata and addenda.
  2. Create typed Pydantic models for each pinned table (like MaterialSpec, JointEfficiency). Each row is a structured record with typed fields.
  3. Write deterministic formula tools (like wall_thickness_cylindrical) that use Decimal arithmetic, accept pinned values as inputs, and return CalculationStep objects with full provenance.
  4. Build a golden evaluation dataset with known-correct answers for calculation queries, so every pinned value and formula can be regression-tested.
  5. Register the standard in the pinned data module so the calculation engine can discover and use the new tables.

This is engineering maturity, not a limitation. Each standard requires ~2–4 weeks of validated data entry + testing. The architecture handles it; the data curation is the gating factor.

Why not just let the LLM extract table values?

Because the failure mode is invisible and catastrophic. An LLM that hallucinates allowable stress S = 138 MPa instead of the correct 114 MPa at 350°C for SM-516 Gr 70 will produce a wall thickness calculation that looks correct but results in a vessel designed 17% thinner than required. No amount of prompt engineering prevents this — the model has no concept of what a "right" stress value is for a temperature it hasn't memorized.

Pinned tables make this failure mode impossible by construction: the LLM never sees the raw numbers until they're already validated and injected by host code.
